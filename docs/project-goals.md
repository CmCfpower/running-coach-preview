# Цели проекта Running Coach

## Главная цель

Создать локальную AI-систему персонального бегового тренера и нутрициолога, которая работает через Telegram, хранит данные пользователя локально, экономно использует LLM и постепенно развивается от простого файлового MVP до полноценной системы с dashboard, аналитикой, расписанием и возможностью деплоя.

## Цели MVP

### 1. Быстро запустить первого полезного бегового бота

Бот должен принимать тренировки, хранить их в понятной файловой структуре, показывать план на день и неделю, учитывать профиль спортсмена и отвечать на базовые вопросы по бегу.

### 2. Не завязываться на Codex

Вся логика должна быть в обычном Python-коде. Инструкции, промпты, схемы и команды запуска должны лежать в репозитории, чтобы проект можно было продолжать в Claude, Qwen или другом агенте.

### 3. Начать с файловой системы

В первой версии не использовать Postgres, Redis, Qdrant, Minio и VDS как обязательные зависимости. Данные должны храниться в `data/` в YAML, JSON и Markdown, чтобы их было легко читать, править и мигрировать.

### 4. Экономить токены

Сырые скриншоты и фото нужно обрабатывать один раз, сохранять структурированный результат и дальше использовать компактные summary, профиль, последние тренировки и текущий план.

### 5. Сделать развитие проекта прозрачным

Вести локальную dashboard-страницу с этапами, сроками, статусами и кратким ТЗ, чтобы было видно, что уже готово, что в работе и когда функционалом можно пользоваться.

## Цели полной версии

### Беговой тренер

- импорт исторических тренировок;
- распознавание скриншотов;
- анализ нагрузки;
- планирование недели;
- утренние напоминания;
- воскресные отчеты;
- корректировка плана после пропусков и признаков усталости.

### Нутрициолог

- прием фото еды;
- примерная оценка калорий и БЖУ;
- рекомендации на остаток дня;
- недельный рацион;
- учет продуктов;
- список покупок.

### Dashboard

Локальная веб-панель должна показывать обзор недели, историю тренировок, тренды, план, питание и inventory.

### Инфраструктура

После стабильного MVP нужные части можно переносить на Postgres, Redis, Qdrant и Minio, добавлять надежное расписание, авторизацию, backups и подготовку к VDS/Vercel.

### Multi-agent architecture

Постепенно выделить специализированных агентов:

- `running-data-agent`;
- `running-plan-agent`;
- `running-schedule-agent`;
- `running-qa-agent`;
- `nutrition-data-agent`;
- `nutrition-plan-agent`;
- `grocery-agent`.

## Критерий успеха первого запуска

Проект считается запущенным в реальную работу, когда локальный Telegram-бот:

- принимает скриншот тренировки;
- сохраняет его в `data/workouts/<date>/raw`;
- умеет читать профиль из `data/profile.yaml`;
- показывает `/today` и `/week`;
- отвечает на простой вопрос по бегу с учетом профиля и текущего плана;
- не требует Codex для ежедневного использования.
