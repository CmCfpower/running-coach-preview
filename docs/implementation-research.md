## 1. Цель

Нужно построить двух Telegram-ботов, которые работают поверх текущей инфраструктуры AI-Infra и используют несколько специализированных агентов.

1. Бот бегового тренера:
   - импортирует исторические скриншоты тренировок из локальной папки;
   - извлекает из них структурированные данные;
   - учитывает профиль спортсмена: возраст, пол, рост, вес, цели, ограничения, пульсовые зоны;
   - составляет тренировочный план в Excel;
   - показывает локальный web-dashboard;
   - принимает новые скриншоты тренировок в Telegram;
   - сохраняет каждую входящую тренировку в папку с датой тренировки;
   - каждое утро присылает план тренировки на день;
   - каждое воскресенье присылает недельную аналитику и план на следующую неделю;
   - отвечает на вопросы по бегу.

2. Бот-нутрициолог:
   - учитывает расписание и результаты тренировок;
   - принимает фотографии еды;
   - оценивает калорийность и БЖУ;
   - дает рекомендации на остаток дня;
   - составляет рацион на день и неделю;
   - отслеживает продукты по спискам покупок;
   - напоминает, что нужно докупить;
   - отвечает на вопросы по питанию.

Система должна экономить токены, хранить исходные данные локально и быть готовой к переносу на VDS.

## 2. Текущая инфраструктура

В AI-Infra уже есть базовые компоненты:

- `agent-hub`: FastAPI-сервис, который принимает задачи и маршрутизирует их к агентам/провайдерам.
- `telegram-bot`: Python Telegram bot runtime с поддержкой proxy.
- `litellm`: OpenAI-compatible proxy для разных LLM-провайдеров.
- `postgres`: долговременное хранение структурированных данных.
- `redis`: очереди, кэш, координация, будущие фоновые задачи.
- `qdrant`: векторное хранилище для семантической памяти и поиска.
- `minio`: S3-compatible хранилище файлов и артефактов.
- `caddy`: reverse proxy для будущего VDS.

Следующее важное улучшение: добавить в `agent-hub` понятие проектов/workspaces. Тогда каждая задача будет иметь project id, локальный путь, границы доступа и набор разрешенных агентов.

## 4. Разделение на агентов

### 4.1 `running-data-agent`

Задачи:

- принимать исторические скриншоты и новые изображения из Telegram;
- сохранять raw-файлы в папки по датам;
- извлекать структурированные поля:
  - дата и время тренировки;
  - дистанция;
  - длительность;
  - средний темп;
  - средний и максимальный пульс;
  - набор высоты;
  - каденс, если виден;
  - субъективная нагрузка, если пользователь добавил текст;
  - источник данных;
- проверять корректность распознавания;
- сохранять нормализованные данные в Postgres и локальный `parsed.json`.

Как экономить токены:

- vision/OCR запускать один раз на каждый скриншот;
- результат сохранять в JSON;
- не отправлять уже разобранные скриншоты модели повторно;
- для известных форматов скриншотов постепенно добавлять детерминированный парсинг.

### 4.2 `running-plan-agent`

Задачи:

- построить первичный план по профилю и истории нагрузок;
- обновлять план после пропусков, тяжелых тренировок или признаков усталости;
- генерировать Excel-файл;
- создавать структурированное назначение на каждый день:
  - тип тренировки;
  - дистанция или длительность;
  - целевой темп;
  - целевая пульсовая зона;
  - разминка и заминка;
  - примечания и предупреждения.

Рекомендуемое поведение:

- начинать консервативно, пока данных мало;
- контролировать рост недельного объема;
- показывать распределение легких/интенсивных тренировок;
- отслеживать признаки усталости: рост пульса на том же темпе, пропуски, резкое увеличение нагрузки.

### 4.3 `running-schedule-agent`

Задачи:

- присылать утренние напоминания о тренировке;
- присылать воскресную аналитику;
- присылать план на следующую неделю;
- замечать пропущенные загрузки тренировок и спрашивать статус.

Варианты реализации:

- MVP: использовать `python-telegram-bot[job-queue]`, где `JobQueue` работает поверх APScheduler.
- Более надежный вариант: отдельный scheduler-сервис с хранением задач в Postgres/Redis, чтобы расписание переживало рестарты контейнеров.

### 4.4 `running-qa-agent`

Задачи:

- отвечать на вопросы по бегу;
- доставать только релевантный контекст:
  - краткий профиль;
  - последние 14 дней;
  - текущий план недели;
  - последнюю недельную сводку;
  - релевантные заметки из Qdrant.

Этому агенту не нужно каждый раз получать всю историю и все скриншоты.

### 4.5 `nutrition-data-agent`

Задачи:

- сохранять фотографии еды по датам и приемам пищи;
- оценивать продукты, калории, белки, жиры, углеводы;
- задавать уточняющие вопросы при низкой уверенности;
- обновлять дневник питания.

Важное ограничение: оценка еды по фото приблизительная. Бот должен давать диапазон и уровень уверенности, а не изображать точность до калории.

### 4.6 `nutrition-plan-agent`

Задачи:

- составлять дневное меню с учетом:
  - профиля;
  - цели;
  - тренировочного расписания;
  - выполненной тренировки;
  - предпочтений;
  - доступных продуктов;
- корректировать рекомендации после фото еды;
- делать недельный рацион по воскресеньям.

### 4.7 `grocery-agent`

Задачи:

- разбирать списки покупок;
- вести остатки продуктов;
- оценивать расход по плану питания;
- напоминать, что докупить.

Для MVP достаточно `inventory.json`, позже можно перенести в Postgres.

## 6. Экономия токенов

Главное правило: сырые скриншоты и фото еды дорогие, структурированные сводки дешевые.

Рекомендуемый pipeline:

1. Принять raw-файл.
2. Один раз запустить vision/OCR.
3. Сохранить структурированный JSON.
4. Сгенерировать компактные дневные и недельные summary.
5. Для дальнейшего reasoning использовать summary, а не исходные изображения.
6. Передавать модели только минимальный релевантный контекст.

Контекст для бегового ответа обычно должен включать:

- краткий профиль спортсмена;
- план текущей недели;
- последние 7-14 дней тренировок;
- недельные агрегаты за последние 6-12 недель;
- вопрос пользователя.

Контекст для питания обычно должен включать:

- профиль питания;
- сегодняшнюю тренировку или план;
- дневник питания за сегодня;
- краткую сводку продуктов;
- вопрос пользователя или JSON, полученный из фото еды.

Чего избегать:

- повторно отправлять всю историю скриншотов;
- повторно отправлять все фото еды;
- тащить длинную неструктурированную историю чата;
- использовать дорогую модель для задач, которые решаются кодом.

Рекомендуемый routing:

- Qwen: модель по умолчанию для cleanup, summary, Telegram-ответов, рутинного планирования и недорогих agent loops.
- DeepSeek: сложный reasoning, критика плана, анализ аномалий, second opinion.
- Vision-capable model: разбор скриншотов и фото еды, если Qwen/DeepSeek недостаточно хорошо справляются с изображениями.
- Claude: ручная или редкая эскалация для сложных ревью, чувствительных объяснений и финальной проверки.
- Детерминированный код: Excel, графики, календарь, сохранение файлов, агрегаты.

## 10. Расписание

MVP schedule:

- ежедневное сообщение по бегу: 08:00;
- ежедневный рацион: 07:30;
- воскресный беговой отчет: 10:00;
- воскресный отчет по питанию и меню на неделю: 11:00;
- напоминания по покупкам: по расписанию или при падении запасов ниже threshold.

Timezone явно фиксируем: `Europe/Moscow`, пока пользователь не изменит.

Для надежности расписание должно храниться в данных, а не только в коде.

## 11. Безопасность и приватность

- `.env` не должен попадать в git.
- `Keys.txt` должен быть в `.gitignore`.
- Сырые fitness/health данные по умолчанию хранятся локально.
- Не отправлять всю историю третьим API без необходимости.
- Для облачных LLM использовать summary и минимальный контекст.
- Добавить границы проекта, чтобы агент не ходил по чужим папкам.
- Перед VDS: HTTPS, авторизация, backups, firewall.

## 13. Claude Pro vs API

Anthropic разделяет Claude-подписки для пользователей и API-биллинг.

Выводы:

- Claude Pro — это подписка для Claude chat experience. По справке Anthropic, Pro не включает API usage через Console. Для API нужен Console account и отдельный API-биллинг.
- Claude Code можно использовать с Pro или Max в терминале. Для этого нужно залогиниться тем же Claude-аккаунтом.
- Использование Claude Code и Claude chat расходует общий лимит подписки.
- Если в окружении задан `ANTHROPIC_API_KEY`, Claude Code может использовать API key вместо подписки, что приведет к API charges.
- Для автономных Telegram-ботов, расписаний и server-side агентов нужен API key, если мы хотим вызывать Claude автоматически.

Практический вывод:

- Claude Pro полезен для ручной разработки через Claude Code.
- Не стоит опираться на Claude Pro как backend для автономных Telegram-ботов.
- Для unattended agents нужны API keys.
- В нашей стратегии Claude не должен быть default-моделью.

## 14. Стратегия моделей

Основная стратегия: Qwen-first.

Используем Qwen:

- как модель по умолчанию;
- для Telegram-ответов;
- для summary;
- для рутинного планирования;
- для недорогих agent loops;
- для предварительного анализа тренировок и питания.

Используем DeepSeek:

- для сложного reasoning;
- для критики тренировочного плана;
- для анализа аномалий;
- для second opinion.

Используем Claude Pro:

- когда пользователь вручную запускает Claude Code в папке проекта;
- для архитектурного ревью;
- для сложных рефакторингов;
- для подготовки промптов и схем;
- для финальной проверки важных решений.

Используем Claude API только если:

- Telegram-бот должен автоматически вызвать Claude;
- нужен дорогой, но качественный final review;
- Qwen/DeepSeek не дают достаточного качества.
