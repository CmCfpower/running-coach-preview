## 1. Назначение проекта

Running Coach - локальная AI-система персонального бегового тренера и нутрициолога. Система должна работать через Telegram, хранить исходные данные локально, экономно использовать LLM и постепенно развиваться от файлового MVP до полноценного набора агентов, dashboard, расписаний и инфраструктурного деплоя.

Главный практический ориентир: сначала получить полезного бегового Telegram-бота на файловой системе, затем добавлять распознавание, аналитику, питание, dashboard и инфраструктуру.

## 2. Подход к реализации

### 2.1 MVP-first

Первая рабочая версия должна быть максимально простой:

- один беговой Telegram-бот;
- хранение данных в файлах;
- минимум внешних сервисов;
- запуск локально;
- переносимость в Claude/Qwen;
- отсутствие обязательной зависимости от Codex.

### 2.2 Filesystem-first

В MVP не использовать как обязательные зависимости:

- Postgres;
- Redis;
- Qdrant;
- Minio;
- Caddy;
- VDS;
- отдельный agent-hub.

Данные должны храниться в `data/` в YAML, JSON и Markdown. Это позволит быстро запускаться, отлаживаться и позже мигрировать в инфраструктуру.

### 2.3 LLM через OpenAI-compatible API

Интеграция с моделями должна идти через настраиваемые параметры:

- `LLM_BASE_URL`;
- `LLM_API_KEY`;
- `LLM_MODEL`;
- optional `LLM_VISION_MODEL`.

Основная стратегия моделей:

- Qwen - default для рутинных ответов, summary, cleanup, простого планирования;
- DeepSeek - сложный reasoning, критика плана, анализ аномалий;
- Claude - ручная разработка, ревью, сложные архитектурные задачи;
- vision-capable model - разбор скриншотов и фото еды.

## 3. Область MVP

### 3.1 Входит в MVP

MVP должен включать:

- структуру проекта и данных;
- `data/profile.yaml`;
- JSON-схемы или Pydantic-модели основных сущностей;
- `.env.example`;
- инструкции для Claude/Qwen-агентов;
- беговой Telegram-бот;
- прием фото тренировок;
- сохранение фото в `data/workouts/<date>/raw`;
- ручное или полуавтоматическое указание даты тренировки;
- хранение `parsed.json` рядом с raw-файлами;
- текущий недельный план в JSON или Markdown;
- команды `/start`, `/profile`, `/today`, `/week`;
- простой QA-flow по бегу на основе профиля, плана и последних тренировок;
- локальная dashboard-страница проекта с roadmap и ТЗ.

### 3.2 Не входит в MVP

В MVP не требуется:

- полноценное OCR/vision-распознавание всех скриншотов;
- Postgres-хранилище;
- Redis-очереди;
- Qdrant-память;
- Minio/S3-хранилище;
- production-деплой на VDS;
- авторизация dashboard;
- отдельный nutrition-бот;
- Excel-отчет;
- полноценная multi-agent orchestration.

Эти пункты должны быть спроектированы так, чтобы их можно было добавить позже без переписывания всей системы.

## 4. Пользовательские сценарии MVP

### 4.1 Первый запуск

Пользователь:

1. Заполняет `data/profile.yaml`.
2. Настраивает `.env`.
3. Запускает Telegram-бота.
4. Пишет `/start`.
5. Получает краткое описание команд.

Критерий приемки:

- бот запускается локально;
- бот отвечает на `/start`;
- отсутствие заполненного профиля обрабатывается понятным сообщением.

### 4.2 Просмотр профиля

Пользователь отправляет `/profile`.

Бот должен:

- прочитать `data/profile.yaml`;
- показать ключевые поля профиля;
- подсказать, какие поля не заполнены.

Критерий приемки:

- команда не падает при пустых/null-полях;
- ответ компактный и читаемый в Telegram.

### 4.3 Загрузка тренировки

Пользователь отправляет скриншот тренировки в Telegram.

Бот должен:

- принять фото;
- определить дату сохранения;
- создать директорию `data/workouts/<date>/raw`;
- сохранить файл с уникальным именем;
- создать или обновить `parsed.json`;
- ответить пользователю, куда сохранена тренировка и что нужно уточнить.

В MVP дата может определяться одним из способов:

- из подписи пользователя;
- из текущей даты;
- через последующую команду уточнения;
- позже - через OCR/vision.

Критерий приемки:

- файл появляется в правильной папке;
- повторная загрузка не перезаписывает предыдущий файл;
- бот сообщает результат.

### 4.4 План на сегодня

Пользователь отправляет `/today`.

Бот должен:

- прочитать текущий план;
- найти назначение на текущую дату в timezone `Europe/Moscow`;
- вернуть тип тренировки, длительность/дистанцию, целевую зону и короткие инструкции.

Критерий приемки:

- при наличии плана команда показывает тренировку;
- при отсутствии плана команда объясняет, что план еще не создан.

### 4.5 План на неделю

Пользователь отправляет `/week`.

Бот должен:

- показать план текущей недели;
- выделить сегодняшний день;
- кратко показать дни отдыха и тренировок.

Критерий приемки:

- ответ помещается в Telegram без чрезмерной длины;
- план читается как расписание.

### 4.6 Вопрос по бегу

Пользователь отправляет текстовый вопрос.

Бот должен:

- собрать компактный контекст;
- вызвать LLM;
- вернуть ответ с учетом профиля и текущего плана;
- не отправлять в LLM всю историю и raw-изображения.

Контекст MVP:

- краткий профиль;
- план текущей недели;
- последние 7-14 дней тренировок, если есть;
- вопрос пользователя.

Критерий приемки:

- текстовый вопрос получает содержательный ответ;
- при недоступности LLM бот возвращает понятную ошибку.

## 7. Модель данных

### 7.1 Профиль спортсмена

Файл: `data/profile.yaml`

```yaml
name: ""
age: null
sex: ""
height_cm: null
weight_kg: null
resting_hr: null
max_hr: null
goals:
  - ""
training_days_per_week: 4
available_training_days:
  - monday
  - wednesday
  - friday
  - sunday
limitations: []
food_preferences: []
allergies: []
```

Требования:

- профиль должен читаться даже при незаполненных полях;
- валидация должна отличать критичные и некритичные пропуски;
- бот должен уметь подсказать, какие поля лучше заполнить.

### 7.2 Запись тренировки

Файл: `data/workouts/<date>/parsed.json`

```json
{
  "id": "2026-05-12-run-001",
  "date": "2026-05-12",
  "source": "telegram",
  "raw_files": ["data/workouts/2026-05-12/raw/telegram_001.jpg"],
  "sport": "running",
  "distance_km": 5.2,
  "duration_sec": 1910,
  "avg_pace_sec_per_km": 367,
  "avg_hr": 148,
  "max_hr": 171,
  "elevation_gain_m": null,
  "cadence_spm": null,
  "rpe": null,
  "extraction_confidence": 0.86,
  "notes": []
}
```

Требования:

- `raw_files` должен хранить относительные пути;
- неизвестные значения должны быть `null`, а не выдуманными;
- `extraction_confidence` обязателен после OCR/vision;
- в MVP допускается ручное заполнение части полей.

### 7.3 План на день

```json
{
  "date": "2026-05-12",
  "type": "easy_run",
  "duration_min": 35,
  "distance_km": null,
  "target_hr_min": 130,
  "target_hr_max": 142,
  "target_pace_min_sec_per_km": 390,
  "target_pace_max_sec_per_km": 430,
  "instructions": "Easy conversational run. Stop if HR drifts above zone for more than 10 minutes."
}
```

Типы тренировок MVP:

- `rest`;
- `easy_run`;
- `long_run`;
- `intervals`;
- `tempo`;
- `recovery`;
- `strength`;
- `cross_training`.

### 7.4 Недельный план

Файл: `data/derived/current_training_plan.json`

```json
{
  "week_start": "2026-05-11",
  "week_end": "2026-05-17",
  "generated_at": "2026-05-12T08:00:00+03:00",
  "summary": "Conservative base week with four runs.",
  "days": []
}
```

`days` должен быть массивом объектов плана на день.

### 7.5 Запись приема пищи

Для полной версии:

```json
{
  "date": "2026-05-12",
  "meal": "breakfast",
  "raw_files": ["data/nutrition/meals/2026-05-12/breakfast.jpg"],
  "items": [
    {"name": "oatmeal", "estimated_grams": 250},
    {"name": "banana", "estimated_grams": 120}
  ],
  "calories_range": [430, 560],
  "protein_g_range": [12, 20],
  "carbs_g_range": [70, 95],
  "fat_g_range": [8, 16],
  "confidence": 0.65
}
```

Оценка питания по фото должна быть диапазоном, а не точным числом.

## 8. Telegram UX

### 8.1 Беговой бот MVP

Команды:

```text
/start
/profile
/today
/week
/plan
/help
```

Дополнительные команды после MVP:

```text
/report
/setdate
/import
/zones
```

Поведение:

- фото без пояснения считается тренировочным скриншотом;
- текст без команды считается вопросом по бегу;
- фразы о пропуске/переносе тренировки должны позже попадать в flow корректировки плана;
- ответы должны быть короткими, Telegram-friendly.

### 8.2 Нутрициолог после MVP

Команды:

```text
/today
/week
/meal breakfast
/groceries
/inventory
/report
```

Поведение:

- фото еды запускает оценку приема пищи;
- список покупок отправляется в grocery-flow;
- вопросы по питанию отвечаются с учетом тренировочного плана.

## 9. LLM и промпты

### 9.1 Промпты

Промпты должны храниться в `prompts/`, а не быть зашитыми в код.

Минимальные промпты:

- `prompts/running_qa.md`;
- `prompts/workout_extract.md`;
- `prompts/running_plan.md`.

### 9.2 Контекст для running QA

В LLM передавать:

- профиль;
- текущий недельный план;
- последние 7-14 дней тренировок;
- последнюю недельную сводку, если есть;
- вопрос пользователя.

Не передавать:

- raw-изображения;
- всю историю тренировок;
- длинную историю чата;
- приватные файлы вне проекта.

### 9.3 Обработка ошибок LLM

Если LLM недоступна:

- бот не должен падать;
- пользователь должен получить понятное сообщение;
- ошибка должна логироваться локально.

## 10. Хранение и файловые операции

### 10.1 Сохранение raw-файлов

Путь:

```text
data/workouts/<date>/raw/<source>_<counter>.<ext>
```

Пример:

```text
data/workouts/2026-05-12/raw/telegram_001.jpg
```

Требования:

- не перезаписывать существующие файлы;
- создавать директории автоматически;
- хранить относительные пути в JSON;
- проверять, что запись идет внутри project root.

### 10.2 Notes

Файл `notes.md` рядом с тренировкой может содержать:

- комментарий пользователя;
- самочувствие;
- субъективную нагрузку;
- уточнения после распознавания.

- сырые health/fitness данные хранить локально;

- передавать внешним API только минимальный контекст;
- ограничивать файловые операции project root;
- перед VDS добавить HTTPS, firewall, auth и backups.

## 14. Переносимость в Claude/Qwen

В проекте должны быть:

- `README.md` с запуском;
- `AGENTS.md` с инструкциями для AI-агентов;
- `CLAUDE.md` с контекстом для Claude Code;

- схемы данных;
- промпты в отдельных файлах;

## 16. Критерий успеха первого реального запуска

Первый запуск считается успешным, если локальный Telegram-бот:

- принимает скриншот тренировки;
- сохраняет его в `data/workouts/<date>/raw`;
- читает профиль из `data/profile.yaml`;
- показывает `/today`;
- показывает `/week`;
- отвечает на простой вопрос по бегу с учетом профиля и текущего плана;
- не требует Codex для ежедневного использования.

- какие форматы скриншотов тренировок наиболее частые;

- нужен ли dashboard на Vercel или достаточно локальной страницы до спортивного dashboard.
